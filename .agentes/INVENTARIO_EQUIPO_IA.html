🎯 SEXTA PARTE — ¿Qué son los "issues"?

📖 Definición simple

Issue = "asunto / problema / cosa que revisar"

En desarrollo de software, cuando un inspector de calidad (QA) revisa el sistema, cada cosa que encuentra para revisar/arreglar se llama UN issue.

Comparación con un auto: Cuando un mecánico inspecciona tu auto, te entrega una lista: "Cambiar bujías", "Hay un ruido en el motor", "Las llantas están gastadas". Cada uno de esos es UN issue.

🚦 Severidad — Qué tan grave es cada uno

ColorNivelSignificadoEjemplo real de hoy
🔴 CRÍTICO Se tiene que arreglar YA. Rompe el sistema, expone datos, o causa pérdida de dinero. Cualquiera con link de tracking veía la ubicación exacta del delivery — riesgo de robo
🟠 ALTO Hay que arreglarlo pronto. Causa problemas serios pero el sistema sigue funcionando. El botón "Pagado" se podía apretar varias veces seguidas y registraba pagos duplicados
🟡 MEDIO Mejorable pero no urgente. Molesta al usuario o complica mantenimiento. Botones con poco contraste de color, difíciles de ver para algunas personas
🟢 BAJO Detalles para pulir. No afectan funcionamiento. Comentarios viejos en el código que ya no aplican

💡 ¿Por qué importa cada issue?

Imaginá tu sistema como una casa:

  • 🔴 Crítico = ventana en la puerta principal abierta — cualquiera entra
  • 🟠 Alto = ventana en planta baja con seguro flojo — con esfuerzo entran
  • 🟡 Medio = ventana arriba sin mosquitero — entran insectos
  • 🟢 Bajo = vidrio rayado — solo se ve feo

Cuando arreglamos issues, vamos cerrando esas ventanas. El sistema se hace más seguro, más rápido y más profesional.

🎓 OCTAVA PARTE — Cómo "integrarte" al equipo (tips para vos)

Para cuando hablemos de equipos, sabé que:

  • "Lanzar un agente" = decirle a una IA que haga una tarea específica
  • "Está corriendo" = está trabajando en este momento
  • "Terminó" = ya entregó su resultado
  • "Se trabó" = se quedó esperando confirmación o tuvo error
  • "Lo relancé" = volví a iniciarlo
  • "En paralelo" = varios trabajando al mismo tiempo
  • "En cascada" = uno detrás del otro

Qué podés hacer vos para apoyar (sin escribir código):

  • Probar el sistema en navegador y reportar cosas raras
  • Tomar decisiones de diseño/negocio cuando te pregunto
  • Configurar secrets en Cloudflare (API keys)
  • Correr SQL en Supabase cuando hace falta
  • Cargar saldo en proveedores de IA
  • Decidir prioridades (qué arreglar primero)

Tu rol más valioso: sos el Product Owner (dueño del producto) — sabés lo que el negocio necesita, los agentes saben cómo escribir el código. Yo soy el puente entre los dos.
